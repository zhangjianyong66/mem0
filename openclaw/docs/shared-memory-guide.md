# 智能共享记忆指南

## 概述

OpenClaw Mem0 插件现在支持**基于大模型的智能共享记忆分类**。系统会自动分析对话内容，智能判断哪些信息应该保存为团队共享记忆。

## 工作原理

### 分类流程

```
对话内容
    ↓
大模型智能分析 (LLM)
    ↓
判断结果：
  ├── 个人记忆 → 保存到 User Scope (代理私有)
  └── 共享记忆 → 保存到 Organization Scope (团队共享)
```

### 大模型 Prompt 设计

系统使用以下标准让大模型判断内容：

**团队角色：** 产品经理、开发工程师、测试工程师、运维工程师

## 适合共享的记忆类型

### 1. 项目架构
- 系统整体架构设计
- 技术栈选型
- 服务拆分方案
- **示例：** "系统采用微服务架构，使用 Kubernetes 部署，分为用户服务、订单服务、支付服务"

### 2. 技术规范
- 代码规范（缩进、命名、注释）
- 代码审查标准
- Git 提交规范
- **示例：** "代码必须使用 ESLint 检查，缩进使用 2 个空格，函数命名使用 camelCase"

### 3. API 设计
- 接口规范
- 请求/响应格式
- 错误码定义
- **示例：** "所有 API 返回统一格式：{code: 200, data: {}, message: 'ok'}，错误码使用 4xx/5xx"

### 4. 环境配置
- 数据库配置
- 中间件配置
- 环境变量说明
- **示例：** "生产环境数据库：PostgreSQL 15，连接池大小 100，超时时间 30s"

### 5. 部署流程
- 发布流程
- CI/CD 配置
- 回滚方案
- **示例：** "发布流程：开发环境 → 测试环境 → 预发环境 → 生产环境，每个环节需人工确认"

### 6. 故障处理
- 常见问题解决方案
- 运维经验
- 监控告警处理
- **示例：** "数据库连接超时，首先检查连接池配置，其次检查网络延迟，最后查看慢查询日志"

### 7. 产品决策
- 需求优先级
- 迭代计划
- 功能范围
- **示例：** "用户管理功能优先级为 P0，本周五上线，包含注册、登录、权限管理"

### 8. 测试标准
- 覆盖率要求
- 测试用例规范
- 验收标准
- **示例：** "核心功能单元测试覆盖率必须 > 80%，接口测试覆盖所有正常和异常场景"

### 9. 监控告警
- 告警规则
- 处理流程
- 联系人
- **示例：** "CPU 使用率 > 80% 持续 5 分钟触发告警，通知运维组，钉钉群：xxx"

### 10. 通用工具和方法
- 开发工具使用
- 调试技巧
- 效率工具
- **示例：** "使用 Docker Compose 本地启动所有服务：docker-compose -f dev.yml up"

## 不适合共享的记忆类型

### 1. 个人主观意见
- "我觉得这个功能不好"
- "我认为应该这样做"
- "我想试试另一种方案"

### 2. 个人帮助请求
- "帮我查一下资料"
- "给我写个脚本"
- "请帮我调试这段代码"

### 3. 临时性内容
- 一次性调试命令
- 针对具体 bug 的讨论
- 临时测试数据

### 4. 个人信息
- 个人偏好设置
- 个人账号密码
- 个人待办事项

## 使用示例

### 场景 1：技术规范讨论

**对话：**
```
开发：我们统一使用 2 空格缩进，禁止使用 tab
开发：ESLint 配置已更新，所有代码必须通过检查
```

**自动分类结果：**
- ✅ 保存到共享记忆 (类别: 规范)
- 原因：这是团队共同遵守的技术规范

### 场景 2：Bug 修复讨论

**对话：**
```
开发：帮我看看这个报错，数据库连接失败了
运维：检查连接池配置，改成 100 试试
开发：好了，谢谢
```

**自动分类结果：**
- ✅ 连接池配置 → 共享记忆 (类别: 故障/环境)
- ❌ "帮我看看" → 私有记忆

### 场景 3：产品需求讨论

**对话：**
```
产品：用户管理功能优先级 P0，本周五上线
开发：收到，我们开始开发
测试：测试用例已准备
```

**自动分类结果：**
- ✅ 需求优先级 → 共享记忆 (类别: 产品)
- 原因：产品决策需要团队同步

## 日志输出

启用智能分类后，日志会显示详细信息：

```
# 旧格式
openclaw-mem0: captured 3 messages (personal: 1, shared: 2)

# 新格式（显示分类类别）
openclaw-mem0: captured 3 messages (personal: 1, shared: 2)
openclaw-mem0: captured 2 shared messages to org [架构, 规范]
```

## 配置说明

系统会自动使用 `config.oss.llm` 中的配置来调用大模型：

```json
{
  "openclaw-mem0": {
    "config": {
      "oss": {
        "llm": {
          "provider": "openai",
          "config": {
            "model": "qwen-max",
            "baseURL": "https://dashscope.aliyuncs.com/compatible-mode/v1",
            "apiKey": "sk-xxx",
            "temperature": 0.1
          }
        }
      }
    }
  }
}
```

**注意事项：**
- 每次对话保存会调用 LLM API，会产生费用
- 如果 LLM 调用失败，会自动降级为规则判断
- 可以通过日志查看分类结果和原因

## 性能优化

### 批量处理
系统会批量发送消息给 LLM 判断，减少 API 调用次数。

### 降级机制
- LLM 调用失败 → 自动使用规则判断
- 网络超时 → 标记为个人记忆（安全策略）

### 缓存策略（未来版本）
- 相同内容直接返回缓存结果
- 相似内容使用向量相似度匹配

## 故障排查

### 问题：所有内容都被标记为个人记忆

**检查：**
1. LLM 配置是否正确
2. API Key 是否有效
3. 查看日志是否有 LLM 调用错误

### 问题：共享记忆包含个人内容

**解决：**
- 这是 LLM 判断误差，可以通过手动删除共享记忆来调整
- 后续对话中更明确地区分"团队规范"和"个人需求"

### 问题：LLM 调用超时

**现象：**
```
openclaw-mem0: captured 3 messages (personal: 3, shared: 0)
# 但日志中有 API timeout 警告
```

**解决：**
- 系统会自动降级为规则判断
- 检查网络连接和 LLM 服务状态

## 最佳实践

1. **明确表达团队规范**
   - 好："团队统一使用 ESLint"
   - 差："我觉得应该用 ESLint"

2. **使用结构化表达**
   - 好："部署流程：测试 → 预发 → 生产"
   - 差："先测试一下，然后发布"

3. **区分个人和团队**
   - 个人："帮我查一下..."
   - 团队："团队规范要求..."

4. **定期整理共享记忆**
   - 使用 `memory_list` 查看共享记忆
   - 删除过时或错误的内容
   - 使用 `memory_search` 验证共享记忆质量
